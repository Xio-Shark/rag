## 正式执行前说明

- 目标：
  - 定位 `codex exec` 在真实 non-dry-run 运行里命中的 DNS / 代理 / endpoint 阻塞来源。
  - 修复当前最小可控范围内的配置或脚本问题，使后续真实短跑不再首先卡在同一外部连通性错误。
  - 保持现有连续任务框架、状态文件和诊断产物兼容。
- 非目标：
  - 不重写连续任务框架。
  - 不修改业务 API、数据库或前端主链路。
  - 不在缺乏证据时盲目替换全部外部端点配置。
- 约束：
  - 优先使用本地事实源定位 `aixj.vip` 配置来源。
  - 若需要改代码，先补测试或至少先固定复现路径。
  - 不暴露敏感凭据到仓库文档或回复中。
- 风险：
  - 若错误来源其实在用户级环境而非仓库脚本，仓库内修改可能无效。
  - 若误改全局 `codex` 配置，可能影响当前环境里的其他任务。
  - 若没有真实短跑验证，只凭静态检查仍可能遗漏运行时继承环境差异。
- 验收标准：
  - 能明确说明 `wss://aixj.vip/responses` 的来源。
  - 若来源在本仓库控制范围内，则完成修复并验证。
  - 若来源在用户环境或外部服务范围内，则完成最小止血和可执行的验证路径，避免继续盲跑。
- 当前项目类型：
  - 主类型：D 线上问题修复
  - 次类型：C 持续任务运行环境排障
- 当前风险等级：
  - R2：涉及持续任务真实运行环境、CLI 配置继承和外部端点连通性。

## 方案与测试计划

- 缺失机制清单：
  - 当前诊断能告诉我们失败类型，但不能直接指出 `codex exec` 继承了哪一层 endpoint / 代理配置。
  - 缺少面向真实环境的最小连通性 smoke 验证步骤。
- 方案对比：
  - 方案 A：直接在仓库脚本里硬改 `codex` 启动参数或环境变量。复杂度低，但风险是误修，且可能覆盖用户环境的真实配置。
  - 方案 B：先定位配置来源，再按最小控制面修复。复杂度略高，但兼容性和回滚性最好。
- 推荐方案：
  - 采用方案 B。
- 是否需要 ADR：
  - 否。本轮以排障和最小修复为主，不引入新基础设施。
- 测试计划：
  - 先确认 `codex` 可执行路径、继承环境变量、配置文件和 `aixj.vip` 来源。
  - 若仓库脚本需要调整，补对应单测并跑目标测试。
  - 若修复落在用户环境，执行一条最小 smoke 命令验证是否还命中相同 endpoint / DNS 错误。
- 实施步骤：
  - 检查仓库内历史文档与本地配置线索。
  - 定位 `codex` / CodexManager / 代理配置来源。
  - 实施最小修复。
  - 复验真实状态和短跑行为。
